# 맡겨멍 — 포트폴리오 기술 문서

## 프로젝트 개요

반려견을 맡길 수 있는 애견 유치원·호텔을 지도 기반으로 탐색하고 예약할 수 있는 iOS 플랫폼.  
공공데이터 동물위탁관리업 API로 전국 업체를 실데이터로 표시하고, 보호자 리뷰·채팅·예약 기능을 통합 제공한다.

**기간:** 2026년 6월 ~  
**역할:** iOS 개발 (Swift/SwiftUI) + Cloudflare Workers 백엔드  
**스택:** Swift 5.9, SwiftUI, Observation, KakaoMapsSDK, TypeScript, Hono, Cloudflare Workers, D1

## 기술적 도전과 해결

### 1. 공공데이터 좌표계 변환

**문제**  
data.go.kr 동물위탁관리업 API가 반환하는 좌표가 WGS84(GPS)가 아닌 EPSG:5181(중부원점 TM) 좌표계라 카카오 지도에 그대로 꽂으면 핀이 엉뚱한 곳에 찍혔다.

**고민**  
외부 라이브러리를 쓰면 간단하지만, iOS 앱에 좌표 변환 라이브러리를 CocoaPods로 추가하면 의존성이 늘어난다. 변환 공식 자체는 수학 연산이라 직접 구현이 가능한 수준이었다.

**해결**  
GRS80 타원체 기반 TM→WGS84 역변환을 `TMConverter` enum으로 직접 구현했다. 변환 후 대한민국 영역(위도 33~39, 경도 124~132)을 벗어나는 좌표는 필터링해 잘못된 핀을 차단했다.

```swift
let coord = TMConverter.toWGS84(x: x, y: y)
guard coord.lat > 33, coord.lat < 39, coord.lon > 124, coord.lon < 132 else { return nil }
```

**이 방법을 선택한 이유**  
외부 의존성 없이 단일 파일(AnimalBoardingService.swift)에서 처리 가능하고, 변환 공식이 국제 표준이라 정확도가 보장된다. CocoaPods 라이브러리 추가 대비 앱 용량도 줄어든다.

### 2. API 키 보안 — xcconfig 분리

**문제**  
`Info.plist`에 공공데이터·카카오·네이버 API 키가 실제 값으로 하드코딩되어 git에 커밋되어 있었다. GitHub에 올리면 누구나 키를 볼 수 있어 무단 사용 또는 쿼터 소진 위험이 있었다.

**고민**  
대안으로 `.env` 파일, 별도 plist, 빌드 스크립트 등이 있었다. iOS에서는 런타임에 환경변수를 읽을 수 없어서 `.env` 방식은 불가능했다.

**해결**  
Xcode의 **xcconfig(Configuration Settings File)** 방식을 채택했다.

- `Secret.xcconfig`에 실제 키값 보관 → `.gitignore`로 git 제외
- `Info.plist`에는 변수 참조만 남김: `$(DATA_GO_KR_SERVICE_KEY)`
- 빌드 시 Xcode가 xcconfig를 읽어 값을 자동 주입

```
// Secret.xcconfig (git 제외)
DATA_GO_KR_SERVICE_KEY = 실제키값

// Info.plist (git 포함)
<key>DATA_GO_KR_SERVICE_KEY</key>
<string>$(DATA_GO_KR_SERVICE_KEY)</string>
```

**이 방법을 선택한 이유**  
Xcode 네이티브 방식이라 추가 코드 없이 기존 `Bundle.main.object(forInfoDictionaryKey:)` 호출을 그대로 유지할 수 있다. Debug/Release 환경별로 다른 키를 주입하는 것도 가능하다.

| 방법 | 장점 | 단점 |
|---|---|---|
| **xcconfig (채택)** | Xcode 네이티브, 추가 코드 없음 | xcconfig 문법 학습 필요 |
| 환경변수 | 간단 | iOS 런타임에서 환경변수 접근 불가 |
| 별도 plist 파일 | 직관적 | 결국 파일 자체를 gitignore 해야 해 xcconfig와 동일 |
| 키를 그냥 커밋 | 편함 | 보안 사고 위험, 키 재발급 필요 |

### 3. Pods git 추적 문제

**문제**  
`.gitignore`가 없어서 CocoaPods의 `Pods/` 디렉토리(279개 파일)가 git에 추적되고 있었다.

**고민**  
이미 git이 추적 중인 파일은 `.gitignore`를 추가해도 자동으로 무시되지 않는다는 점을 인지해야 했다.

**해결**  
`.gitignore` 생성 후 `git rm -r --cached Pods/`로 추적 목록에서만 제거했다. 실제 파일은 삭제되지 않고 git이 더 이상 관리하지 않는다.

**이 방법을 선택한 이유**  
iOS 생태계 표준 관례다. `Podfile.lock`만 커밋하면 누구든 `pod install` 한 번으로 동일한 버전을 재현할 수 있다.

### 4. 하드코딩 사용자 정보 제거 — UserProfile 모델

**문제**  
`BookingView`의 보호자 이름·연락처·주소, `HomeView`의 인사말이 문자열로 박혀 있었다.

```swift
formField(label: "이름", value: "김상민")   // 하드코딩
Text("상민님")                               // 하드코딩
```

**고민**  
Core Data나 Keychain을 쓰면 과하다. 간단한 사용자 설정값이라 UserDefaults로 충분하다고 판단했다.

**해결**  
`@Observable` 기반 `UserProfile` 모델을 만들어 `UserDefaults`에 영속 저장하고 `RootView`에서 환경 객체로 주입했다.

```swift
@Observable
final class UserProfile {
    var name: String {
        didSet { UserDefaults.standard.set(name, forKey: "profile_name") }
    }
    // ...
}
```

뷰에서는 `@Environment(UserProfile.self)`로 읽어 사용한다.

**이 방법을 선택한 이유**  
Swift Observation 프레임워크를 사용해 `@Observable`로 선언하면 변경 시 관련 뷰가 자동으로 업데이트된다. `didSet`으로 UserDefaults에 즉시 저장해 앱 재시작 후에도 값이 유지된다.

### 5. TypeScript any 타입 제거

**문제**  
Cloudflare Workers 백엔드의 모든 요청 바디가 `any` 타입으로 처리되고 있어 TypeScript를 쓰는 의미가 없었다.

**해결**  
라우트별 요청 바디 타입을 명시적으로 정의했다.

```typescript
type ReservationBody = {
  user_id?: number; userId?: number
  start_date?: string; startDate?: string
  // ...
}

const body = await c.req.json<ReservationBody>()  // any → 명시적 타입
```

**이 방법을 선택한 이유**  
iOS 클라이언트에서 snake_case와 camelCase를 혼용해 보내는 구조가 이미 있었기 때문에 두 형태 모두 optional로 받는 타입을 정의했다. 잘못된 구조의 요청이 들어오면 개발 중에 타입 오류로 바로 잡을 수 있다.

### 6. 예약과 채팅방의 도메인 설계 — 공공 가게의 ID 부재 문제

**문제**  
예약(`reservations`)은 `store_id`가 필수인데, 지도의 가게는 공공데이터 API에서 오기 때문에 우리 DB에 `store_id`가 없다(이름·주소만 존재). 또 예약할 때마다 채팅방을 새로 만들면, 같은 가게에 두 번 예약한 사용자가 서로 다른 방을 갖게 되어 대화가 흩어진다.

**고민**  
- **가게 식별**: (A) 예약마다 임시 번호 부여 — 같은 가게를 못 묶음. (B) 이름만 저장 — 구조적 식별 불가. (C) 공공 가게를 우리 `stores`에 등록해 안정적 번호 부여.
- **채팅방 범위**: 방을 '예약'에 묶을지, '가게'에 묶을지, '(사용자+가게)'에 묶을지. 다중 사용자를 고려하면 (사용자+가게)라야 남의 대화가 섞이지 않는다.

**해결**  
- 리뷰 기능에서 이미 쓰던 `store_key`("이름\|주소") 방식을 재사용. 예약 시 `store_key`로 `stores`를 조회해 **있으면 재사용, 없으면 등록(자동 증가 번호)** → 같은 가게는 항상 같은 `store_id`.
- 채팅방은 `chat_rooms(user_id, store_id UNIQUE)` 테이블로 **(사용자+가게) 조합당 1개**. 예약 시 해당 방을 조회/생성하고 그 방에 '예약 요청 완료' 메시지를 남긴다.
- API가 `{reservation_id, room_id}`를 함께 반환해 클라이언트가 이후 채팅방으로 바로 연결할 수 있게 했다.

```typescript
// 같은 가게는 같은 번호, 같은 (유저·가게)는 같은 방
const storeId = existing ? existing.store_id : (await insertStore(...)).id
const roomId  = room     ? room.room_id     : (await insertRoom(userId, storeId)).id
```

**이 방법을 선택한 이유**  
공공 API 데이터를 우리 도메인으로 끌어올 때 흔한 "외부 식별자 ↔ 내부 식별자" 매핑 문제다. `store_key` upsert로 외부 데이터를 자연스럽게 내부 엔티티로 승격시키고, 리뷰와 동일한 키를 써서 일관성을 유지했다. (사용자+가게) 방 설계는 카카오톡 채널 방식과 동일해 사용자에게 직관적이다.

### 7. 첫 화면이 첫 탭에 반응하지 않던 버그 — SwiftUI AttributeGraph 순환

가장 오래 붙잡았던 버그이자, SwiftUI 내부 동작을 깊이 이해하게 된 사례다.

**문제 (증상)**
앱을 새로 설치하고 **처음 켠 시작 화면**에서 역할 선택 카드(보호자 / 보호자·사장님)를 탭하면, 체크 표시도 테두리 색도 전혀 바뀌지 않았다. 그런데 한 번 로그인했다가 **로그아웃해서 다시 나타난 "똑같은" 시작 화면**에서는 멀쩡히 동작했다. 즉 **"콜드 런치 직후 첫 진입"에서만** 나타나는 비대칭 버그였다.

**왜 어려웠나**
탭 자체는 정상이었다. 로그를 심어 보니 탭할 때마다 상태값(`selectedRole`)은 `.owner`로 잘 바뀌었고, 그 상태로 로그인하면 실제로 사장님 계정으로 로그인까지 됐다. **상태는 바뀌는데 화면(`body`)만 그 변화를 다시 그리지 않는** 기묘한 상황이었다. 처음에는 표면적인 원인들 — 뷰 전환 애니메이션, 버튼 스타일, 상태를 저장하는 위치, 빌드 캐시 — 을 하나씩 의심하고 수정했지만 전부 빗나갔다.

**체계적 진단**
추측을 멈추고 사실을 좁혀 나갔다.

1. `body` 진입점에 로그를 심어, 탭 후 **`body`가 재평가 자체를 안 한다**는 것을 확정했다. (렌더링만 얼어붙은 게 아니라 아예 다시 계산되지 않았다.)
2. 시뮬레이터를 명령줄에서 직접 실행해(`xcrun simctl launch --console-pty`) 콘솔을 캡처했더니 결정적 로그가 나왔다:

```
=== AttributeGraph: cycle detected through attribute 5436 ===
```

**근본 원인**
SwiftUI는 뷰의 의존 관계를 **AttributeGraph**라는 방향 그래프로 관리하고, 어떤 값이 바뀌면 그 그래프를 따라 영향받는 뷰를 다시 그린다. 이 그래프에 **순환(cycle)** 이 생기면 SwiftUI는 무한 루프를 막기 위해 **그 경로의 갱신 전파를 끊어버린다.** 그 결과가 바로 "처음 한 번만 그려지고 이후로는 어떤 상태 변화에도 얼어붙는" 화면이었다.

순환의 근원은 시작 화면의 상단 여백 코드였다:

```swift
.padding(.top, UIApplication.safeAreaTop + 12)   // safeAreaTop이 UIKit 윈도우의 safeAreaInsets를 읽음
```

`safeAreaTop`은 UIKit 윈도우의 `safeAreaInsets`를 **`body` 평가 도중에** 읽는다. 그런데 콜드 런치 순간엔 윈도우 레이아웃(safe area)이 아직 확정되기 전이라, 다음과 같은 순환이 만들어진다:

```
화면 콘텐츠 크기  →(의존)  body 평가  →(읽음)  safe area  →(의존)  윈도우 레이아웃  →(의존)  콘텐츠 크기 …
```

로그아웃 후 재진입 때는 윈도우가 이미 자리를 잡아 safe area가 확정돼 있으므로 순환이 생기지 않는다 — 이것이 "콜드 런치에서만 실패하는" 비대칭의 정체였다.

**해결 (가장 효과적이었던 방법)**
핵심은 **`body` 평가 중에 UIKit 레이아웃 값을 읽지 않는 것**이다. safe area 값을 `@State`에 캐시하고, 레이아웃이 끝난 뒤 실행되는 `onAppear`에서 실제 값을 채워 넣도록 바꿨다. `body`는 이 `@State` 값만 읽으므로 UIKit 레이아웃과의 의존 고리가 끊어져 순환이 사라진다.

```swift
private struct SafeAreaTopPadding: ViewModifier {
    @State private var topInset: CGFloat = 56          // 기본 safe area(44) + 12
    func body(content: Content) -> some View {
        content
            .padding(.top, topInset)                    // body는 캐시된 @State만 읽음 (순환 없음)
            .onAppear { topInset = UIApplication.safeAreaTop + 12 }  // 레이아웃 확정 후 실제 값 주입
    }
}
```

처음에는 시작 화면에만 인라인으로 적용해 원인을 검증했고, **같은 위험 패턴이 다른 11개 화면에도 있었기 때문에** 재사용 `ViewModifier`(`.safeAreaTopPadding()`)로 묶어 `SafeAreaKey.swift` 한 곳에서 관리하도록 정리했다. 이제 `body`에서 safe area를 직접 읽는 곳은 0곳이다.

**검증**
수정 전/후를 콘솔 캡처로 비교했다.

| | 순환 감지 | 탭 후 body 재평가 |
|---|---|---|
| 수정 전 | **1건** | ❌ 안 됨 (화면 얼어붙음) |
| 수정 후 | **0건** | ✅ 됨 (`topInset` 56→59로 갱신되는 로그 확인) |

**이 방법을 선택한 이유**
다른 선택지도 검토했다.

| 방법 | 문제점 |
|---|---|
| RootView의 `.ignoresSafeArea()`를 걷어내고 SwiftUI 기본 safe area 사용 | 12개 화면의 레이아웃을 전부 바꿔야 해 범위·위험이 큼 |
| `GeometryReader`로 safe area 읽기 | RootView가 safe area를 무시하는 구조라 0을 반환 → 사용 불가 |
| **`@State` 캐시 + `onAppear` (채택)** | 기존 레이아웃·아키텍처를 그대로 두고 "`body`에서 UIKit 읽기"만 제거하는 **최소·국소 수정** |

`@State` 캐시 방식은 근본 원인("`body`에서 레이아웃을 읽어 순환이 생김")을 정확히 겨냥하면서 화면 배치나 앱 구조를 전혀 건드리지 않는다. 게다가 `ViewModifier`로 묶어 재발 방지와 코드 중복 제거(DRY)까지 함께 달성했다.

**배운 점**
- **"상태는 바뀌는데 화면이 안 바뀐다"** 는 SwiftUI 특유의 증상은 AttributeGraph 순환을 의심할 강한 신호다.
- `body`는 순수해야 한다. 평가 도중 UIKit 레이아웃처럼 "그 결과가 다시 `body`에 영향을 주는" 값을 읽으면 순환이 생길 수 있다. safe area 같은 값은 캐시하거나 SwiftUI 네이티브 수단으로 받아야 한다.
- UI 자동화 없이도, `body`에 심은 로그 + `simctl` 콘솔 캡처만으로 "추측"이 아닌 "관측"에 근거해 원인을 특정하고 수정을 검증할 수 있었다.

### 8. 사장님↔손님 양방향 채팅 — 역할을 '플래그'에서 '계정'으로

**문제**  
채팅이 사실상 단방향이었다. 손님이 가게에 문의를 보내면 DB에 쌓이기만 할 뿐, 사장님이 그 메시지를 보거나 답장할 방법 자체가 없었다(백엔드에 '가게 주인' 개념이 없음). 더 근본적으로는, 시작 화면에서 고른 역할(보호자/사장님)이 **같은 계정의 플래그**(`isOwner`)일 뿐이어서 두 역할이 같은 `user_id`를 공유했다 — 채팅·찜·예약·최근 본 가게가 전부 섞이고, 사장님으로 보낸 메시지조차 `sender_id`가 같아 손님이 보낸 것처럼(오른쪽 말풍선) 표시됐다.

**고민**  
- **방 구조**: 사장님용 문의방을 따로 만들 것인가? → 만들면 같은 대화가 두 방으로 갈라진다. 기존 "(손님+가게)당 1방" 설계를 지키고 사장님이 **같은 방에 참여**하는 쪽이 맞다.
- **역할 분리 수준**: 플래그를 유지한 채 화면만 사장님 시점으로 바꿔줄 수도 있지만, `sender_id`가 같은 이상 눈속임일 뿐이다. 역할을 **계정 단위로 분리**해야 데이터가 구조적으로 나뉜다.
- **가게 소유 연결**: 지도의 가게는 공공데이터 출신이라 "이 가게 주인이 누구인지"를 서버가 알 방법이 없다. 소유 관계를 어디에, 어떤 UX로 만들 것인가.

**해결**  
1. **방 재사용**: 사장님용 방을 만들지 않고, `stores.owner_id`(마이그레이션)로 가게 소유를 연결한 뒤 `GET /api/owners/:id/chatrooms`로 "내 가게로 온 방 목록"을 조회해 기존 채팅 목록 상단에 '받은 문의' 섹션으로 노출했다. 말풍선이 원래 `sender_id == 내 userId` 기준이었기 때문에, **채팅방 화면은 한 줄도 고치지 않고** 사장님 시점(사장님 오른쪽/손님 왼쪽)이 자동으로 성립했다.
2. **역할의 계정 귀속**: `users.is_owner` 컬럼을 신설해 역할을 **최초 가입 시 카카오 계정에 영구 귀속**시켰다. 같은 계정으로 반대 역할 로그인을 시도하면 서버가 409로 거절하고 앱이 안내 문구를 띄운다. 모든 데이터가 `user_id` 기준이므로 별도 마이그레이션 없이 채팅·찜·예약이 역할별로 자동 분리됐다. 시뮬레이터 개발자 진입도 역할별 계정(`dev-simulator` / `dev-simulator-owner`)으로 나눠, 기기 한 대에서 로그아웃→역할 전환만으로 양쪽을 테스트할 수 있다.
3. **가게 소유 등록**: 마이페이지 '내 가게'에서 상호명으로 공공데이터 업체를 검색해 등록한다(`POST /api/stores/claim` — 리뷰·예약과 동일한 `store_key` upsert 재사용, 남의 가게면 409, 잘못 등록 대비 해제 API 포함). 받은 문의·받은 예약 요청 모두 이 소유 관계로 스코프된다.
4. **실시간감**: 방을 열어둔 동안 3초 폴링으로 상대 메시지를 반영한다. WebSocket(Durable Objects)이 정석이지만, 포트폴리오 범위에서는 "열려 있는 방"에만 도는 폴링이 비용 대비 충분했다.

**이 방법을 선택한 이유**  
핵심은 "새 기능을 위해 새 구조를 만들지 않는다"였다. 방 재사용은 §6의 도메인 설계(조합당 1방)를 지켰고, 가게 등록은 기존 `store_key` upsert 패턴을 그대로 썼으며, 말풍선 시점 분리는 이미 있던 `sender_id` 비교 로직이 계정 분리의 부수 효과로 공짜로 얻어졌다. 역할을 계정으로 승격한 것도 UI 분기 대신 **식별자(user_id) 층위에서 문제를 풀어** 이후 기능들이 자동으로 올바르게 동작하게 한 결정이다.

### 9. 백엔드 의존성 취약점 해소 — 고장난 로컬 npm 환경을 sudo 없이 우회

**문제**  
`npm audit`에서 백엔드 의존성 취약점 6건(high 5·low 1)이 보고됐다. 특히 **hono는 배포 번들에 실제로 실리는 유일한 런타임 의존성**인데 path traversal·CORS 와일드카드 반사 등 5건의 권고가 걸려 있었다(나머지 undici·ws·esbuild는 wrangler 개발 도구 체인). 그런데 `npm audit fix`를 실행할 수 없었다 — 과거 `sudo npm install`의 흔적으로 `node_modules`와 `package-lock.json`이 **root 소유**라, 일반 사용자 권한의 npm이 lockfile과 node_modules 어느 쪽에도 쓰지 못하고 EACCES로 실패하는 상태였다.

**고민**  
- **정석은 `sudo chown`**이지만 비밀번호 입력이 필요한 대화형 작업이라, 가능하면 사용자 개입 없이 풀고 싶었다.
- **POSIX 권한의 세부를 이용한 우회**: 파일의 삭제·이름 변경 권한은 파일 자체가 아니라 **부모 디렉터리의 쓰기 권한**을 따른다. 부모(`backend-cloudflare/`)는 내 소유이므로, root 소유 `package-lock.json`도 "읽어서 사본 → 삭제 → 사본 복원" 순서로 소유권을 탈환할 수 있었다.
- 같은 논리로 `node_modules`도 이름을 바꿔 치우려 했지만 **실패** — 디렉터리 이동은 그 디렉터리 내부의 `..` 항목 갱신이 필요해 디렉터리 자체의 쓰기 권한을 요구하고, macOS는 같은 부모 안에서의 rename조차 거부했다. 파일은 되고 디렉터리는 안 되는 비대칭.

**해결**  
`node_modules`를 치울 수 없다면 그 자리에서 고치는 것을 포기하고, **깨끗한 임시 디렉터리에 프로젝트의 매니페스트만 복사해 작업**했다:
1. `package.json` + `package-lock.json`을 임시 디렉터리로 복사 → `npm install`(npm 캐시 일부도 root 소유라 `npm_config_cache`로 캐시 위치까지 우회) → `npm audit fix` → **6건 전부 0건으로**.
2. 결과는 hono 4.6.10→4.12.30, wrangler 4.93.0→4.110.0 — 모두 기존 `^` 범위 내 semver 호환 갱신이라 `package.json`은 그대로이고 **lockfile만 변경**됐다.
3. 검증도 그 자리에서: `src/`와 `wrangler.toml`을 임시 디렉터리로 복사해 갱신된 wrangler로 **직접 배포**했다. 배포 대상은 `wrangler.toml`의 워커 이름으로 결정되므로 저장소 밖에서 실행해도 같은 워커에 반영된다. 배포 성공(cron 트리거 유지 확인) 후 읽기 전용 엔드포인트 3종을 curl로 스모크 테스트했다.
4. 갱신된 `package-lock.json`만 저장소로 복사해 커밋. 저장소의 root 소유 `node_modules`는 건드리지 않았고(git 추적 대상도 아님), 근본 수리는 `sudo chown` 1회로 가능함을 기록해뒀다.

**이 방법을 선택한 이유**  
lockfile 갱신은 그 자체로는 아무것도 바꾸지 않는다 — Cloudflare Workers는 **배포 시점에 로컬 node_modules를 번들**하므로, "새 의존성으로 설치된 환경에서의 재배포"까지가 한 세트여야 hono 갱신이 실제 런타임에 반영된다. 임시 디렉터리 방식은 이 세트를 한곳에서 완결시키면서(설치→audit→배포→검증) 고장난 로컬 환경과 저장소를 전혀 건드리지 않았다. 환경이 깨져 있을 때 "환경부터 고치고 작업"이 아니라 **작업에 필요한 최소 환경을 옆에 새로 만드는** 접근이 개입 범위도, 실패 시 되돌릴 것도 가장 적었다.

### 10. 알림장(diary) 기능 — 새 구조를 만들지 않고 기존 도메인 위에 얹기

**문제**  
사장님이 맡은 강아지가 하루를 어떻게 보냈는지 보호자에게 전하는 '알림장'이 필요했다. 어린이집 알림장처럼 사장님이 글을 남기면 보호자가 읽는 흐름이다. 문제는 이 기록을 **무엇에 귀속시킬지**가 애매하다는 것이었다 — 가게? 예약? 채팅방?

**고민**  
- **귀속 대상**: (A) 가게에 붙이면 여러 예약 건이 한 타임라인에 섞인다. (B) 채팅방에 붙이면 잡담과 기록이 뒤엉킨다. (C) **예약에 붙이면** "이번에 맡긴 그 건"의 기록만 시간순으로 모여, 보호자가 '이번 예약은 어땠지'를 바로 본다.
- **작성/열람 화면**: 사장님(작성)과 보호자(열람)에 별도 화면 두 개를 만들 것인가? 두 화면은 데이터도 레이아웃도 거의 같고 '쓰기 UI 유무'만 다르다.
- **알림 전달**: 사장님이 글을 남긴 걸 보호자가 어떻게 아는가? 새 알림 채널(푸시)을 만들 것인가?

**해결**  
세 고민을 모두 기존 구조 재사용으로 풀었다.
1. **예약 귀속 · 타임라인**: 알림장을 `reservations`에 1:N으로 묶고, 한 예약에 엔트리가 시간순으로 쌓이는 타임라인으로 설계했다(`GET /api/reservations/:id/diaries`). 보호자는 예약 내역 카드의 '알림장 보기'로, 사장님은 홈 FAB → 확정 예약 목록에서 진입한다.
2. **화면 1개, 시점 2개**: `DiaryTimelineView` 하나를 `router.diaryContext.canWrite` 플래그로 공유한다 — 사장님이면 하단 '알림장 쓰기' 바와 각 카드의 수정/삭제 메뉴가 켜지고, 보호자면 읽기 전용이다. §8의 채팅 말풍선과 같은 "같은 화면, 플래그로 시점 분리" 패턴이다.
3. **알림은 채팅 자동 메시지로**: 사장님이 알림장을 **작성**하면(수정·삭제는 제외) 서버가 그 보호자의 채팅방에 자동 메시지(sender 0, '맡겨멍')를 남긴다 — `"${강아지이름}의 새 알림장이 도착했어요\n예약 내역에서 확인해주세요"`. 예약 취소·리뷰 요청 안내에 이미 쓰던 sender 0 패턴을 그대로 재사용했고, 안 읽은 채팅 빨간 점 로직에 자동으로 얹혀 **별도 알림 채널이 필요 없었다**.

강아지 이름을 실제로 보여주려고 예약 조회에 `pets` 조인을 더했다(`LEFT JOIN pets … p.name AS pet_name`) — 값이 없으면 '우리 아이'로 폴백해 화면이 비지 않게 했다.

**사진은 스키마만 먼저**  
알림장에 사진이 붙으면 좋지만, 이미지 저장은 Cloudflare R2 활성화가 필요해 이번 범위에서 뺐다. 다만 나중에 마이그레이션을 다시 쪼개지 않도록 `diary_photos` 테이블은 이번에 미리 만들어 두고(마이그레이션 0011), API는 빈 배열을 반환하게 했다. 기능이 준비되면 업로드/서빙 엔드포인트와 클라이언트 `PhotosPicker`만 붙이면 된다.

**이 방법을 선택한 이유**  
새 기능일수록 새 구조를 세우고 싶은 유혹이 크지만, 알림장은 이미 있는 세 축(**예약 귀속 · 역할 플래그 · sender 0 자동 메시지**) 위에 얹는 것으로 충분했다. 덕분에 채팅방 화면과 알림 로직은 한 줄도 건드리지 않았고, 새로 쓴 코드는 '타임라인 뷰 + diary CRUD 라우트'로 국한됐다.

### 11. 이모지를 에셋 이미지로 — 인라인 렌더링의 벽

**문제**  
앱 곳곳(빈 상태 안내, 섹션 제목, 채팅 자동 메시지 등)에 유니코드 이모지(🐾 📅 💬 …)를 그대로 썼다. 이모지는 OS·폰트 버전에 따라 모양이 달라지고 브랜드 톤과도 어긋나, 디자인 통일을 위해 자체 PNG 에셋으로 교체하기로 했다.

**해결(1) — 폴백형 EmojiIcon**  
모든 이모지가 에셋을 가진 건 아니라, "에셋이 있으면 이미지, 없으면 이모지 텍스트"로 자동 폴백하는 `EmojiIcon` 뷰를 만들었다. 이모지 문자열 → 에셋 이름 매핑(`petAssetName`)을 한 곳에 모으고, 프레임 크기를 받아 그린다.

```swift
if let name = emoji.petAssetName {
    Image(name).resizable().scaledToFit().frame(width: size, height: size)
} else {
    Text(emoji).font(.system(size: size))   // 에셋 없으면 이모지 그대로
}
```

**부딪힌 벽 — 흐르는 텍스트 속 이모지는 크기를 못 맞춘다**  
로그인 화면 '따뜻한 손길로…' 옆의 발바닥처럼, 문장 중간에 작은 에셋을 끼워 넣고 싶었다. `Text("…\(Image(asset))")`로 인라인 합성했더니 **이미지가 본문 폰트 크기(~17pt)로 커져** 문구를 압도했고, 그 런에 `.font(.system(size: 12))`를 걸어도 비트맵 이미지는 폰트 크기를 따르지 않아 줄어들지 않았다. 여러 방식으로 시도했지만 인라인 합성된 비트맵은 신뢰성 있게 축소되지 않았다.

**해결(2) — 인라인은 포기하고 규칙을 세움**  
결론은 "에셋 이모지는 **프레임 크기를 가진 독립 요소로만 쓴다**"였다. 문장 중간의 인라인 이모지는 전부 제거하고(로그인 푸터·자동 메시지 발바닥 포함), 크기 제어가 필요한 곳은 `HStack { Text(…); EmojiIcon(…) }`처럼 이미지를 별도 요소로 분리했다. 에셋이 아예 없는 이모지(📭 👤 ✍️ 📝 🙏 등)는 톤이 깨지느니 **텍스트만 남기고 지웠다** — 있는 것만 이미지, 없는 것은 글자, 어중간한 혼용은 없앤다는 규칙.

**배운 점**  
SwiftUI의 `Text` 인라인 이미지는 SF Symbol처럼 **폰트 메트릭을 따르는 벡터**에는 잘 맞지만, 고정 크기 비트맵 PNG에는 맞지 않는다. 아이콘을 문장에 자연스럽게 흘려 넣어야 한다면 PNG가 아니라 SF Symbol이나 폰트 글리프를 썼어야 했다. 크기 제어가 필요한 비트맵은 레이아웃 요소로 분리하는 것이 맞다.

### 12. 마스킹된 주소 정밀화 — 이관된 네이버 API를 추적해 연결하기

**문제**  
공공데이터 동물위탁관리업 API의 주소는 개인정보 보호로 상세 번지가 마스킹되어 온다 — `"서울특별시 양천구 목동중앙남로 **, ***호 (목동)"`. 가게 상세 화면은 카카오 키워드 검색으로 전체 주소를 보강하고 있었지만, 일부 가게는 여전히 마스킹 주소가 그대로 노출됐다.

**원인 조사 — 카카오 하나로는 안 되는 이유**  
앱과 동일한 API 호출을 curl로 재현해 보니(키는 xcconfig에서 셸 변수로만 로드, 값 미출력) 문제 가게들은 **카카오맵에 아예 등록돼 있지 않아** 키워드 검색이 0건이었다. 검색 반경을 늘리는 우회는 더 위험했다 — 이름이 비슷한 **다른 지역 가게**가 잡힌다(실측: 동명 가게가 47km 밖에서 매칭). 좌표→주소 변환 폴백도 써봤지만, 핀 좌표의 지번 주소일 뿐 사업자 등록 주소(도로명 + 층·호)와 달라 "맞지 않는 주소"가 됐다.

**벽 — 네이버 검색 API가 조용히 이사 갔다**  
네이버 플레이스에는 이 가게들이 있어서 네이버 지역 검색 API로 전환하기로 했는데, 발급받은 키가 계속 인증 실패(024)를 냈다. 키 문제로 단정하지 않고 엔드포인트 자체를 의심해 조사한 결과: **developers.naver.com의 검색 API는 신규 제휴를 받지 않고, NCP(네이버클라우드) NAVER API HUB로 이관**된 상태였다. 새 키는 형식부터 다르고(ID ~10자/Secret ~40자), 엔드포인트는 `naverapihub.apigw.ntruss.com/search/v1/local`, 헤더는 `X-NCP-APIGW-API-KEY-ID`/`-KEY`를 쓴다. 다행히 **응답 JSON은 구 API와 동일**해서, 엔드포인트와 헤더만 바꾸면 기존 파싱 코드가 그대로 동작했다 — 지역 검색(`NaverLocalService`)과 블로그 후기(`NaverBlogService`) 둘 다 이 방식으로 연결했다.

**해결(1) — 화면 로직: 네이버 우선 + 좌표 검증**  
가게 상세의 보강 체인을 재구성했다.
1. **주소는 네이버 우선** — 네이버 플레이스의 등록 주소는 `"목동중앙남로 80 동구햇살APT 2차 제1층 제107호"`처럼 층·건물·호까지 담고 있다(카카오 도로명은 번지까지만).
2. **전화·카카오맵 링크는 카카오** — 네이버 지역 검색의 telephone 필드는 항상 비어 오므로(하위호환 잔재) 두 API를 `async let` 병렬 호출해 역할을 나눴다.
3. **둘 다 없으면 좌표→지번 폴백** — 최소한 동·번지는 보이게.
4. **오매칭 방어** — 검색 결과 중 핀 좌표 최근접 후보를 고르고 **3km 밖이면 기각**한다. 또 네이버 주소는 끝에 상호명이 붙어 오는 경우가 있어(`"백마로 487 멍스테이"`) 꼬리표를 제거했다.

**해결(2) — DB 백필: 표시용 컬럼만, 자릿수 검증으로**  
찜 목록·내 가게처럼 DB의 `stores.address`를 그대로 쓰는 화면을 위해, 검증된 실주소를 DB에도 백필했다. 좌표가 없는 가게는 **마스킹 패턴 자릿수 대조**로 검증했다 — 마스킹의 `*` 하나는 숫자 한 자리이므로 `인천타워대로***번길 **`를 정규식 `인천타워대로\d\d\d번길 \d\d`로 바꿔 후보 주소와 대조하고 시군구까지 일치해야 통과. 이 기준으로 20곳 중 16곳을 백필했고, 네이버·카카오 어디에도 근거가 없는 4곳은 마스킹을 유지했다(추측으로 채우지 않는다).

**불변식 — 정체성과 표시의 분리**  
여기서 지킨 규칙 하나: `store_key`(`"이름|마스킹 주소"`)는 찜·예약·리뷰·채팅이 전부 매달린 **정체성 키라서 절대 건드리지 않고**, 표시용 `address` 컬럼만 갱신했다. 백엔드 upsert가 기존 행의 address를 덮어쓰지 않는 것도 확인해(phone·좌표만 COALESCE 보강) 백필 값이 이후 앱 사용으로 되돌아가지 않음을 보장했다.

**배운 점**  
외부 API는 지식이나 문서가 낡을 수 있다 — 인증 실패를 "키가 잘못됐다"로 단정하지 않고 엔드포인트의 존속 자체를 의심한 것이 돌파구였다. 그리고 처음부터 정체성 키와 표시용 데이터를 분리해 둔 설계(§6) 덕분에, 주소를 통째로 갈아 끼우는 백필이 다른 도메인에 아무 파급 없이 안전했다.

### 13. 푸시 알림 — 원격 푸시 없이 "알림처럼 동작하는" 기능 만들기

**문제**  
유료 Apple Developer Program 멤버십이 없어 APNs(원격 푸시)를 쓸 수 없다. 그런데 채팅 메시지·예약 확정·취소 같은 이벤트는 사용자가 앱을 보고 있지 않을 때도 알려야 한다.

**고민**  
- 멤버십 결제 전까지 기능 자체를 미룰지, 대체 수단을 찾을지.
- 대체 수단 후보: (A) Firebase Cloud Messaging — 하지만 iOS에서는 FCM도 결국 내부적으로 APNs 인증서를 거치므로, 유료 멤버십이 없으면 이름만 다를 뿐 똑같은 벽에 부딪힌다. (B) 서버가 만든 통합 이벤트 피드를 클라이언트가 주기적으로 조회해 로컬 알림으로 변환.
- (B)로 결정한 뒤에도 두 문제를 더 풀어야 했다 — 같은 이벤트가 두 번 알림으로 뜨지 않게 하는 **중복 방지**, 그리고 앱이 화면 밖에 있을 때도 최소한의 커버리지를 주는 **백그라운드 갱신**.

**해결(1) — 서버 통합 알림 피드 + opaque 커서**  
`GET /api/users/:id/notifications`가 계정별 커서 이후의 새 이벤트만 내려준다. 커서는 클라이언트가 내용을 해석하지 않고 문자열 그대로 저장했다가 다음 요청에 그대로 돌려주는 **opaque 커서**로 설계해, 서버의 커서 포맷이 나중에 바뀌어도 클라이언트 코드는 안 건드려도 되게 했다.

**해결(2) — 이벤트당 정확히 한 축, 중복 방지 불변식**  
앱에는 이미 '알림처럼 보이는' 통로가 하나 있었다 — 알림장 도착·사장님 취소 안내·리뷰 요청 같은 이벤트는 채팅방에 자동 메시지(sender 0)로 이미 남고 있었다. 여기에 새 알림 피드까지 같은 이벤트를 넣으면 사용자는 알림을 두 번 받는다. 규칙을 하나 세웠다: **이미 채팅 메시지를 남기는 이벤트는 채팅 축으로만 노출하고, 메시지가 없는 이벤트(새 예약 요청·확정·고객 취소)만 알림 피드(예약 축)에 넣는다.** 코드 몇 줄로 강제되는 게 아니라 "새 이벤트 종류를 추가할 때마다 지켜야 하는 규칙"이라 `CLAUDE.md`에 설계 불변식으로 명시해뒀다.

이 불변식이 실제로 성립하는지는 백엔드의 `INSERT INTO chat_messages` 호출 지점을 전부 추적해, "채팅 메시지가 남는 이벤트(채팅 축)"와 "안 남는 이벤트(확정·고객취소 — 예약 축)"가 겹침도 빠짐도 없이 나뉘는지 검증하는 방식으로 확인했다.

**해결(3) — 클라이언트: 포그라운드 폴링 + BGAppRefreshTask**  
- 앱이 켜져 있을 때는 30초 주기로 피드를 조회해 로컬 알림(`UNUserNotificationCenter`)으로 발행한다.
- 앱이 백그라운드에 있을 때는 `BGAppRefreshTask`로 iOS가 짬을 낼 때 같은 폴링을 한 번 수행한다.
- 알림을 탭하면 이벤트 종류에 따라 채팅방 / 예약 내역 / 받은 예약 요청 중 하나로 딥링크한다.

**받아들인 한계 — "실시간"이 아니다**  
`BGAppRefreshTask`는 서버가 능동적으로 기기를 깨우는 원격 푸시와 근본적으로 다르다. 앱이 백그라운드에 남아있는 동안 **iOS가 재량으로** 짬을 내 실행해줄 뿐이라 실행 시점을 전혀 보장할 수 없고, 사용자가 앱 스위처에서 완전히 스와이프해 종료하면 이 메커니즘 자체가 멈춘다(원격 푸시는 종료 상태에서도 온다). 이 격차는 유료 멤버십 없이는 코드로 메울 수 없는 순수한 플랫폼 제약이라, `PLAN.md`에 "원격 푸시는 유료 멤버십 확보 후 v2"로 명시하고 지금 방식을 그 안에서 낼 수 있는 최선의 대체제로 받아들였다. 같은 초에 여러 이벤트가 겹치는 커서 경계 유실, 취소된 예약 요청이 커서가 지나가면 다시 안 보이는 것 같은 엣지 케이스도 함께 — 실사용 트래픽에서 충돌 확률이 낮고, 완전히 막으려면 별도 이벤트 로그 테이블이 필요해 이번 범위엔 과했다.

**어려웠던 부분 — 알림 델리게이트의 async 브리징 버그**  
전체 기능 중 재현·원인 파악이 가장 까다로웠던 버그다.

증상: 알림 센터에서 알림을 탭해 앱을 열고 곧이어 화면 안의 버튼을 누르면 `NSInternalInconsistencyException: Call must be made on main thread`로 앱이 즉시 종료됐다.

코드만 보면 이상이 없었다 — `UNUserNotificationCenterDelegate`의 `didReceive` 구현은 `nonisolated func ... async`였고, 내부에서 `await MainActor.run { ... }`로 정확히 메인 액터에 진입해 상태를 바꾸고 있었다. Swift 동시성 규칙만 놓고 보면 올바른 코드였다.

원인은 Swift 코드 자체가 아니라 **그 코드가 호출되는 경로**에 있었다. `UNUserNotificationCenterDelegate`의 `async` 버전 메서드는 내부적으로 Objective-C의 completion-handler 기반 프로토콜에 자동으로 연결(bridging)되는데, 이 브리징 스텁은 "콜백은 호출된 바로 그 스레드(메인 스레드)에서 동기적으로 와야 한다"고 가정한다. 함수 안에서 `await`로 한 번이라도 스레드를 넘나들면(`MainActor.run`으로 다시 메인으로 돌아오더라도) 재개(resume) 시점이 원래 스레드와 미묘하게 어긋나 이 가정이 깨지고, 런타임이 앱을 강제 종료시킨다.

스택 트레이스의 mangled 심볼(`$s16Dog_kindergarden22AppNotificationServiceC...didReceive...`)을 실제 함수명으로 되짚어 `didReceive` 프레임임을 확정했고, 해결은 **애초에 async를 쓰지 않는 것**이었다. 델리게이트 메서드를 전통적인 completion-handler 형태로 되돌리고, 메인 액터로의 전환은 `await`가 아니라 `Task { @MainActor in ... }`로 fire-and-forget 처리한 뒤 `completionHandler()`는 그 자리에서 동기 호출하도록 바꿨다. 함수 자체가 동기 함수가 되므로 문제의 브리징 스텁을 아예 타지 않는다.

**배운 점**  
Swift 동시성 관점에서 '정석'으로 보이는 코드(`async` 델리게이트 + `await MainActor.run`)가 항상 안전한 건 아니다 — 그 프로토콜이 원래 Objective-C 세계(completion handler)에서 왔다면, Swift가 자동 생성한 브리징 레이어의 스레드 가정까지 함께 신경 써야 한다. 정적 타입 검사와 컴파일 성공은 이런 런타임 계약 위반을 잡아주지 못했다 — 시뮬레이터에서 "알림 탭 → 앱 안에서 조작"이라는 실사용 시나리오를 손으로 재현해야만 드러나는 문제였다.

**어려웠던 부분 — "맞게 설계했지만 틀린" 알림 히스토리**  
알림 센터 identifier를 처음엔 방(room) 단위로 고정했다(`"chat-\(roomId)"`) — 같은 방에서 메시지가 여러 번 와도 알림 센터에 방마다 하나만 남도록 **의도적으로** 설계한 것이었다(알림 센터가 같은 대화로 도배되는 걸 막으려는 목적이었고, 리뷰 단계에서도 스펙 위반은 아니었다).

문제는 실제로 써보고서야 드러났다. "안녕" "하세요"를 연달아 보내면 "하세요 외 1건"으로 알림 센터에 잘 뜬다. 그런데 한참 뒤에 "바이"를 보내면, 같은 identifier라 새 알림이 **이전 알림을 지우고 그 자리를 대체**해버려 "하세요 외 1건"을 확인하지 못한 채 영영 사라졌다. "방마다 알림 1개로 깔끔하게 유지"라는 원래 의도가, 사용자 입장에선 "못 본 알림이 조용히 사라진다"는 정반대 경험이 된 것이다. identifier를 방 단위 고정에서 매번 고유한 값(UUID)으로 바꿔, 알림이 서로 대체되지 않고 각각 쌓이도록 고쳤다.

**이 방법을 선택한 이유**  
"스펙대로 구현됨"과 "실제로 써보니 맞음"은 다른 질문이라는 걸 확인한 사례였다. 코드 리뷰·빌드 성공만으로는 못 잡고, 시뮬레이터에서 두 계정을 오가며 실제 알림 흐름을 손으로 재현해야만 발견되는 종류의 문제였다.

## 아키텍처 결정

### iOS — SwiftUI + Observation

UIKit 대신 SwiftUI를 선택한 이유:
- `@Observable` 매크로로 보일러플레이트 없이 상태 관리 가능
- 선언형 UI로 지도·리뷰·예약 화면을 빠르게 구현
- iOS 17+ 기준 Observation이 `@StateObject`/`@ObservedObject` 대비 성능이 뛰어남

### 백엔드 — Cloudflare Workers + D1

Spring Boot 대신 Cloudflare Workers를 선택한 이유:
- 서버 관리 불필요, 글로벌 엣지 배포 자동화
- D1(SQLite)으로 마이그레이션 파일 기반 스키마 관리 가능
- 포트폴리오 배포 비용 무료

### 지도 데이터 — 공공데이터 API

카카오·네이버 플레이스 대신 공공데이터 동물위탁관리업 API를 선택한 이유:
- 정부가 인허가한 업체만 등록되어 있어 신뢰도 높음
- 전국 데이터를 한 번에 가져올 수 있어 초기 로딩 후 필터링이 용이
- 마스킹된 주소는 네이버(층·건물 상세)→카카오(전화·링크)→좌표 변환 순의 다중 폴백으로 보강해 정확도 향상 (§12)

## UI/UX 설계 고민

디자인 시안을 기능으로 옮기며 "정보를 어떻게 배치하고, 사용자의 마찰을 어떻게 줄일까"를 계속 고민했다.

### 강아지 카드 — 정보 밀도 조절

한 줄에 `이름 / 나이 / 성별 / 견종`을 슬래시로 나열하니 답답했다. **이름을 카드의 주인공으로 키우고**, 나이는 이름 옆에, 성별·견종은 아랫줄로 분리해 시선이 자연스럽게 흐르도록 했다. 사회성·알레르기·특이사항 같은 세부 정보는 목록에서 빼고, **카드를 탭하면 열리는 상세 시트**에서 항목별 칸으로 보여줬다 — 목록은 가볍게, 상세는 충분하게.

```
[🐶]  몽이 3살          ← 이름(강조) + 나이
      남 · 포메라니안    ← 성별 · 견종
```

### 한 카드 안의 두 동작 — 탭 영역 분리

강아지 카드에는 '상세 열기'와 '삭제(−)' 두 동작이 공존한다. 카드 본문은 `contentShape(Rectangle())`로 탭 영역을 명확히 잡아 상세 시트를 열고, 삭제 버튼은 그 바깥의 독립 버튼으로 두어 오작동(삭제하려다 상세가 열리는 등)을 막았다.

### 지도 — '전국보기'를 없애고 '보이는 만큼만'

기존엔 도(道) 단위로 전환하는 '전국보기' 모드가 있었는데, 사용자가 도 경계를 신경 써야 해서 부자연스러웠다. 지도를 움직이면(`cameraDidStopped`) **현재 화면의 위경도 범위를 계산해 그 안의 가게만** 도 경계와 무관하게 표시하도록 바꿨다. 핀이 과밀해지는 걸 막으려 화면 중심에서 가까운 순으로 개수 상한을 뒀다.

### 예약 화면 — 실제 데이터와의 마찰 줄이기

- **강아지 선택**을 실제 등록 데이터로 연결해, 고르면 나이·견종·몸무게·메모가 함께 갱신된다.
- **보호자 연락처·주소**는 한 번 입력하면 `UserDefaults`에 영구 저장되고, 다음 예약부터 '자동 불러오기'로 재사용한다. 매번 다시 적는 마찰을 없앴다.
- **서비스를 고르면 총 결제금액이 즉시 다시 계산**된다(가격 문자열 파싱 + 픽업 고정액 합산). 완료화면의 '결제 예정'도 같은 값으로 통일했다.

### 로딩·빈 상태·실패를 눈에 보이게

목록이 비면 안내 문구, 통신 중엔 스피너 + 버튼 비활성, 실패 시 빨간 문구로 상태를 드러냈다. 사용자가 '지금 무슨 일이 일어나는지' 알 수 있게 하는 데 신경 썼다.

### 향후 UI 개선 아이디어

- 강아지 아바타에 실제 사진 업로드(현재는 기본 이미지 순환)
- 예약 날짜를 텍스트 목록 대신 캘린더 피커로, 호텔은 체크인/아웃 기간 선택
- 지도 핀 클러스터링(밀집 지역 묶기)
- 채팅 실시간화(WebSocket) 및 읽음 표시
